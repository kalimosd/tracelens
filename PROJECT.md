# TraceLens

## 项目定位

TraceLens 是一个面向流畅度与响应性问题初分析的 Perfetto Trace Agent。

它服务的核心角色是基础体验组和性能工程师：输入本地 trace、补一句场景描述、可选指定重点 process，系统自动定位异常时间窗口，识别关键线程角色，串联证据链，输出初步归因和优化方向，再把结果交给业务团队继续落地修改。

这个项目不是通用聊天工具，也不是泛化 Agent 平台。它的核心价值是：
- 基于 Perfetto trace 做结构化取证
- 输出可复核的分析链，而不是黑盒结论
- 沉淀成可复用的分析能力，而不是一次性 prompt

## MVP 定义

### 目标用户
- 基础体验组
- 性能工程师
- 需要使用该工具的团队成员

### 第一版要解决的问题
针对交互类、状态切换类的流畅度问题，基于本地 Perfetto trace 快速完成一次可信的初步诊断。

这里的场景包括但不限于：
- 滑动卡顿
- 页面切换卡顿
- 模式切换卡顿
- 进入页面或切换状态后的响应性下降

第一版不把场景写死成 scrolling，而是聚焦“交互流畅度分析”。

### 第一版输入
- 本地 trace 文件
- 一句场景描述
- 可选重点 process，例如 `com.example.launcher`

### 第一版输出
- 分析结论
- 关键证据
- 完整或尽量完整的分析链
- 初步优化方向

第一版不直接生成业务代码修改方案，它的职责是完成基础体验组的“初分析 + 证据整理 + 优化方向建议”。

### 第一版交付形态
- CLI 内核：用于开发、调试、批量验证、能力沉淀
- 简单 Web 壳：用于团队试用和后续推广

### 第一版交互方式
- 默认单轮分析：一次输入，一次给出完整结果
- 支持后续多轮追问：围绕已有结果继续下钻

## 核心设计原则

### 1. Skill-first 的混合 Agent
常见分析路径优先走 Skill，保证稳定、可复核、可沉淀。

Agent 负责理解问题、安排分析步骤、组合证据；只有在 Skill 不足时，才进入受控 SQL 探索。

### 2. 分层架构优先
不把产品交互、Agent 编排、分析能力、框架适配、trace 查询混在一起。

第一版宁可少做一点能力，也要先把边界划清楚，保证后续可扩展。

### 3. 以业务 process 为主视角
分析默认围绕用户指定的重点 process 展开；如果未指定，再由系统推断。

在此基础上，关联关键系统线程和依赖进程，而不是默认做全局无边界扫描。

### 4. 强可解释性
结果不能只给一句判断。

需要展示：
- 异常窗口是怎么定位出来的
- 关键线程角色是怎么识别的
- 哪些 Skill 或查询提供了核心证据
- 最终结论是如何收敛出来的

### 5. 不把语义绑定死在 Android
第一版可以先优先验证 Android 主路径，但架构上不能把线程模型和分析逻辑写死在 Android 特有语义里。

系统需要预留对 Flutter 框架相关线程角色和渲染链路的适配能力，例如 UI、Raster 等角色识别与分析。

## 系统架构

### 1. 产品交互层
职责：
- 接收本地 trace 上传
- 接收场景描述和重点 process
- 展示结论、证据、分析链
- 支持结果基础上的继续追问

组成：
- CLI 入口
- 最小可用 Web 页面

### 2. Agent 编排层
职责：
- 理解用户场景
- 生成轻量分析计划
- 决定先查异常窗口还是先识别线程角色
- 选择并编排 Skill
- 在必要时触发受控 SQL 探索
- 将离散证据收敛成结论和优化方向

Agent 是分析指挥层，不是把所有领域知识都塞进 prompt 的黑盒。

### 3. 分析能力层
职责：
- 异常窗口定位
- 线程角色识别
- process / 线程关联分析
- Skill 执行
- 受控 SQL 探索
- 证据整理与分析链生成
- 初步优化方向生成

这一层是产品最核心的能力资产。

### 4. Trace 语义适配层
职责：
- 把不同运行时的 trace 语义映射成统一分析语言
- 识别主线程、RenderThread、Flutter UI、Raster、binder 线程、系统关键线程等角色
- 识别不同运行时下的调度、渲染、阻塞模式

这一层用于吸收 Android 与 Flutter 等不同运行时的差异，避免上层分析逻辑被具体框架绑死。

### 5. 数据访问层
职责：
- 封装 trace_processor 的加载与查询
- 标准化查询结果和元数据
- 为 Skill、自由 SQL、artifact store 提供统一边界

上层模块不直接散写底层 SQL 访问逻辑。

## 一次完整分析流程

### 1. 输入归一化
接收：
- 本地 trace 文件
- 场景描述
- 可选重点 process

系统先将输入整理成统一的分析上下文。

### 2. 初始理解与分析计划
Agent 首先：
- 判断这是哪类流畅度问题
- 生成轻量分析计划

例如：先定位异常窗口，再识别关键线程，再检查阻塞链和调度异常。

### 3. 异常窗口定位
如果用户没有给出时间范围，系统应自动从 trace 中寻找最可能异常的时间窗口。

候选信号包括：
- 帧节奏异常
- 长 slice
- 线程长时间不可运行或被阻塞
- 调度延迟
- 明显的响应性异常区间

### 4. 主视角建立
围绕重点 process 建立主分析视角。

如果用户填写了 process，例如 `com.example.launcher`，则优先围绕该 process 识别关键线程角色；如果未填写，则由系统推断主视角对象。

### 5. Skill-first 取证
优先调用稳定 Skill 获取证据，例如：
- 线程状态分布
- 长任务识别
- 调度延迟分析
- 阻塞链分析
- 渲染节奏异常检测
- 跨进程关键依赖摘要

如果 Skill 不足以解释问题，再进入受控 SQL 探索。

### 6. 归因收敛
Agent 需要将分散证据收敛成结构化结论：
- 卡顿点在哪里
- 主要影响线程或进程是谁
- 根因更像调度问题、阻塞问题、渲染问题还是依赖链问题
- 哪些证据支持这个判断
- 哪些地方还存在不确定性

### 7. 结果输出
第一版输出至少包含四部分：
- 结论摘要
- 关键证据
- 分析链
- 初步优化方向

### 8. 多轮追问
在单轮分析结束后，保留关键 artifact 和查询历史，用于后续追问，例如：
- 为什么怀疑这个线程
- 这个窗口内还有哪些异常
- 再展开看某条阻塞链

## 第一版能力清单

第一版明确包含：
- 本地 trace 上传
- 场景描述输入
- 可选重点 process 输入
- 异常窗口定位
- 线程角色识别
- 以重点 process 为主视角的分析
- 与关键系统线程和依赖进程的关联分析
- 一批核心 Skill
- 可解释的分析链输出
- 初步优化方向生成
- 基于已有结果的多轮追问能力

## 第一版优先 Skill 类型

建议第一批优先沉淀以下 Skill：
- 异常窗口候选检测
- 重点 process 关键线程发现
- 线程状态分布分析
- 长 slice / 长任务识别
- 调度延迟与 runnable 堆积分析
- 阻塞链 / 等待链分析
- 渲染节奏异常检测
- 跨进程关键依赖摘要

## 第一版非目标

第一版不追求：
- 覆盖所有 Perfetto 分析场景
- 直接生成业务侧修复代码
- 一开始就接入复杂采集平台
- 默认执行全局无限制跨进程分析
- 完全依赖 LLM 自由生成 SQL

## 路线图

### Phase 1：CLI 内核与主分析链
目标：先跑通最小可用分析能力。

包含：
- trace_processor 数据访问封装
- 最小 Agent 编排循环
- 第一批核心 Skill
- 异常窗口定位
- 线程角色识别
- 终端可解释输出

### Phase 2：最小 Web 壳
目标：让团队可以开始试用。

包含：
- 本地文件上传
- 场景描述与 process 输入
- 结构化结果展示
- 复用 CLI 内核能力

### Phase 3：多轮追问与 Artifact 复用
目标：让单次分析结果能继续下钻。

包含：
- artifact store
- 查询历史复用
- 结果导航与摘要增强

### Phase 4：适配层与场景扩展
目标：提升框架兼容性和分析覆盖面。

包含：
- 更多 Skill
- 更强的线程角色识别
- 更完善的 Flutter 框架相关语义适配
- 更广的流畅度问题场景支持

### Phase 5：团队化使用能力
目标：支撑更广泛的内部使用。

包含：
- 更好的报告导出
- case 复盘与共享
- 后续可选的平台接入

## 验收标准

第一版成功的标准不是“完全自动化”，而是：

基础体验组或性能工程师可以基于这个工具输出的证据链，完成一次可信的初步诊断，并给出初步优化方向，再交给业务团队继续落地。

换句话说，第一版要证明的是：它已经能成为真实工程流程中的“第一分析入口”。
